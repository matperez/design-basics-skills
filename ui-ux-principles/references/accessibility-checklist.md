# Accessibility & Inclusive Design — чеклист

## Зачем

- 15% мирового населения (1+ млрд человек) имеют ту или иную форму ограничений.
- 1 из 12 мужчин и 1 из 200 женщин имеют нарушения цветовосприятия.
- 4% населения — с нарушениями зрения, 0.6% — слепые.
- Accessibility улучшает опыт для **всех**, не только для людей с ограничениями.

## Спектр ограничений (Microsoft Inclusive Design)

| Тип | Постоянное | Временное | Ситуативное |
|-----|-----------|-----------|-------------|
| **Зрение** | Слепота | Травма глаза, операция | Яркое солнце, ночной режим |
| **Слух** | Глухота | Ушная инфекция | Шумное место |
| **Моторика** | Одна рука | Сломанная рука | Держит ребёнка |
| **Когнитивное** | Дислексия, ADHD | Стресс, усталость | Multitasking, спешка |

## WCAG 2.1 — минимальный чеклист

### Восприятие (Perceivable)

- [ ] **Контраст текста** ≥ 4.5:1 (обычный текст), ≥ 3:1 (крупный текст ≥ 18pt / 14pt bold)
- [ ] **Контраст UI-элементов** ≥ 3:1 (кнопки, inputs, иконки)
- [ ] **Не только цветом** — информация передаётся ещё иконкой, текстом или паттерном
- [ ] **Alt text** — все изображения имеют описание (или `alt=""` для декоративных)
- [ ] **Масштабирование** — контент читаем при zoom 200%
- [ ] **Captions** — видео имеют субтитры; аудио — транскрипцию
- [ ] **Адаптация** — контент не теряется при смене ориентации или размера экрана

### Управление (Operable)

- [ ] **Keyboard** — всё доступно с клавиатуры (tab, shift+tab, enter, escape, arrow keys)
- [ ] **Focus visible** — активный элемент чётко выделен (outline или ring)
- [ ] **Skip links** — «Skip to content» для screen reader пользователей
- [ ] **Touch targets** — минимум 44×44px (Apple HIG) / 48×48dp (Material)
- [ ] **Spacing** — достаточное расстояние между touch targets, чтобы избежать ошибочных нажатий
- [ ] **No keyboard traps** — пользователь может выйти из любого элемента
- [ ] **Timing** — нет ограничений по времени, или можно продлить
- [ ] **Motion** — respect `prefers-reduced-motion`; без мигающего контента (≤ 3 вспышек/сек)

### Понимание (Understandable)

- [ ] **Labels** — все формы имеют видимые labels (не только placeholder)
- [ ] **Ошибки** — описательные сообщения + предложение исправления
- [ ] **Язык** — простой, понятный; указан `lang` для HTML
- [ ] **Предсказуемость** — навигация консистентна; неожиданных изменений нет
- [ ] **Помощь** — contextual help доступна там, где сложно

### Надёжность (Robust)

- [ ] **Semantic HTML** — правильные теги (`<nav>`, `<main>`, `<button>`, `<h1>`–`<h6>`)
- [ ] **ARIA** — labels, roles, states для кастомных компонентов (но semantic HTML > ARIA)
- [ ] **Testing** — проверено screen reader'ом (VoiceOver, NVDA, JAWS)

## Типы нарушений цветовосприятия

| Тип | Что не различают | Распространённость |
|-----|-----------------|-------------------|
| **Protanopia** | Красный-зелёный (красный ослаблен) | Самый частый |
| **Deuteranopia** | Красный-зелёный (зелёный ослаблен) | Частый |
| **Tritanopia** | Синий-жёлтый | Редкий |
| **Achromatopsia** | Все цвета (только оттенки серого) | Очень редкий |

**Как проверить:** используй плагины-симуляторы (Color Blind, Able) или DevTools → Rendering → Emulate vision deficiencies.

## Практические советы

1. **Дизайн сначала в grayscale** — если иерархия работает без цвета, она будет работать и с ним.
2. **Проверь без мыши** — пройди весь flow только клавиатурой.
3. **Включи screen reader** — послушай, как звучит твой интерфейс.
4. **Accessibility ≠ отдельная задача** — это часть каждого шага дизайна, не финальный чеклист.
5. **Узнай у пользователей** — лучший источник правды о доступности — люди с ограничениями.

## Aesthetic-Usability Effect

Красивый интерфейс воспринимается как более удобный, даже если это не так. Это не повод игнорировать usability, а напоминание: хороший визуальный дизайн **усиливает** хороший UX, но не заменяет его.
