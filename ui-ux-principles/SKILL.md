# UI/UX Design Principles — процесс от проблемы до валидированного интерфейса

Этот скилл — пошаговый процесс: от определения проблемы и аудитории до готового, протестированного интерфейса. Не справочник «что такое UX», а дерево решений для агента.

---

## Шаг 1. Определить проблему и аудиторию

Прежде чем проектировать интерфейс, зафиксируй фундамент.

### 1.1. Этика и ответственность (до пикселей)

| Вопрос | Зачем |
|--------|-------|
| Какую **реальную проблему** решаем? | Не изобретаем ещё один weather app |
| Кому можем **навредить**? | Тёмные паттерны, манипуляция, утечка данных — избегать |
| Какие **предубеждения** несём? | Unconscious bias влияет на дизайн; состав команды должен отражать аудиторию |
| Где данные пользователя, кто их видит? | Прозрачность и контроль над данными — с первого экрана |

**Правило:** если дизайн-решение приносит пользу бизнесу за счёт пользователя — пересмотри решение.

### 1.2. Бриф

Уточни у клиента (или выведи из контекста):

| Элемент | Что фиксируем |
|---------|---------------|
| **Problem statement** | Одно предложение: кто + что не может + почему |
| **Целевая аудитория** | Демография, контекст, техническая грамотность |
| **Персоны** (1–3) | Имя, цель, боль, сценарий использования |
| **Бизнес-цели** | Что значит «успех» для бизнеса? |
| **Ограничения** | Бюджет, сроки, платформы, существующая дизайн-система |
| **Метрики** | По чему поймём, что решили проблему? |

### 1.3. Design Thinking — 3 линзы для каждой идеи

| Линза | Вопрос |
|-------|--------|
| **Desirability** | Это нужно пользователю? |
| **Feasibility** | Мы можем это построить? |
| **Viability** | Это жизнеспособно для бизнеса? |

Сильная идея — на пересечении всех трёх.

## Шаг 2. Исследовать

Исследование отвечает на вопрос «Что на самом деле делают и чувствуют пользователи?»

### Два типа данных

| Тип | Что даёт | Методы |
|-----|----------|--------|
| **Quantitative** (что делают) | Цифры, паттерны поведения | Analytics, A/B тесты, eye tracking, surveys (закрытые) |
| **Qualitative** (почему делают) | Мотивации, эмоции, контекст | Интервью, usability tests, дневники, наблюдение |

### Методы по фазе проекта

| Фаза | Метод | Что получаем |
|------|-------|-------------|
| **Strategy** | User personas | Понимание, для кого строим |
| **Strategy** | Интервью | Глубокие инсайты о болях и целях |
| **Strategy** | Card sorting | Как пользователи группируют информацию (→ IA) |
| **Design** | Task analysis | Валидация: может ли пользователь выполнить задачу? |
| **Design** | Eye tracking | Куда смотрят, что пропускают |
| **Design** | Surveys | Восприятие: удобно ли, приятно ли |
| **Launch** | A/B testing | Какой вариант работает лучше |
| **Launch** | Analytics | KPI: время на странице, bounce, клики |
| **Post-launch** | Usability bug testing | Баги UX, которые снижают качество |
| **Post-launch** | Дневники | Как продукт вписывается в жизнь |

Подробнее: `references/research-methods-matrix.md`.

### Принцип: исследуй рано, исследуй часто

- Фидбек на этапе wireframe = дешёвый фикс.
- Фидбек после запуска = дорогой фикс.
- AI может помочь структурировать большие объёмы фидбека, но не заменяет прямой контакт с пользователями.

## Шаг 3. Спроектировать структуру (IA и потоки)

Прежде чем рисовать UI, определи **что где находится** и **как пользователь перемещается**.

### Information Architecture (IA)

| Компонент | Что | Как проверить |
|-----------|-----|---------------|
| **Organization** | Группировка контента по смыслу | Card sorting с пользователями |
| **Labeling** | Названия разделов и кнопок | Понятны без объяснений? |
| **Navigation** | Меню, tabs, breadcrumbs | Может ли пользователь найти нужное за 3 клика? |
| **Search** | Фильтры, подсказки, результаты | Для больших продуктов — обязательно |

### Принципы простоты

- **Прогрессивное раскрытие** — не показывай всё на одном экране. Каждый шаг решает одну задачу.
- **Знакомые паттерны** — используй существующие paradigms (hamburger menu, tab bar, card layout). Не изобретай заново то, что пользователи уже знают.
- **Правило 3–5 тапов** — основное действие достижимо за 3–5 взаимодействий.
- **FUD-фильтр** (Fear, Uncertainty, Doubt) — если интерфейс вызывает тревогу (финансы, здоровье, долгосрочные решения), упрости ещё сильнее.
- **Белое пространство** — пустота направляет внимание. Перегруженный макет = ничего не выделяется.

### User flows

Для каждого ключевого сценария нарисуй путь (подробный процесс — в скилле `prototype-wireframe`).

## Шаг 4. Применить визуальное восприятие

Теперь, когда структура есть, применяй законы восприятия, чтобы пользователь **видел главное, группировал правильно и действовал быстро**.

### 4.1. Gestalt — 6 ключевых принципов для UI

| Принцип | Правило для дизайна | Пример |
|---------|-------------------|--------|
| **Proximity** | Связанные элементы — рядом; несвязанные — разделить пространством | Иконка + подпись = одна единица |
| **Similarity** | Одинаковая форма/цвет = одна группа | Кнопки одного уровня — одинаковый стиль |
| **Continuity** | Выравнивание ведёт взгляд | Навигация, прогресс-бары, таймлайны |
| **Closure** | Мозг достраивает неполные формы | Минималистичные иконки, логотипы с вырезами |
| **Figure-Ground** | Передний план чётко отделён от фона | Модальные окна с затемнённым фоном |
| **Focal point** | Один уникальный элемент = одна точка внимания на экране | CTA контрастного цвета |

**Тест:** закрой глаза → открой → что увидел первым? Это и есть твоя иерархия. Если первое — не CTA, перестрой.

Подробнее с 11 принципами и чеклистом: `references/gestalt-principles.md`.

### 4.2. Визуальная иерархия — 7 инструментов

| # | Инструмент | Как работает |
|---|-----------|-------------|
| 1 | **Alignment** | Выровненные элементы = связанные. Сетка — фундамент. |
| 2 | **Color** | Яркость и насыщенность задают порядок считывания |
| 3 | **Contrast** | Различие рядом усиливается; контраст ≥ 4.5:1 для текста (WCAG) |
| 4 | **Proximity** | Близкие элементы = группа. Чанкинг: блоки по 3–5 элементов |
| 5 | **Size** | Крупное — первым. Размер влияет и на accessibility |
| 6 | **Texture** | Тактильная подсказка о функции (акценты без цвета) |
| 7 | **Time** | Анимация, переходы, progressive disclosure |

Подробнее: `references/visual-hierarchy.md`.

### 4.3. Закон Фитца — размещение элементов

**MT = a + b × log₂(2D / W)** — время до цели зависит от расстояния (D) и ширины (W).

| Правило | Применение |
|---------|-----------|
| Крупнее + ближе = быстрее | CTA — крупная, в зоне активности |
| **Prime pixel** — вероятная позиция курсора | Основные действия — рядом |
| **Magic pixels** — углы экрана | Второстепенное: настройки, logout |
| **Thumb zone** (mobile) | CTA — центр / низ экрана; tab bar внизу |
| Padding вместо размера | Расширяй область клика padding'ом, не увеличивая визуально |

Подробнее с иллюстрациями: `references/fitts-law.md`.

## Шаг 5. Применить 7 принципов UI

Когда переходишь к визуальному слою, каждый принцип — чеклист.

| # | Принцип | Что проверить |
|---|---------|-------------|
| 1 | **Hierarchy** | Самое важное видно первым? Пользователь знает, где он (как в книге: заголовок → глава → секция)? |
| 2 | **Progressive disclosure** | Информация порционирована? Есть индикатор прогресса (шаги, процент)? |
| 3 | **Consistency** | Одинаковые компоненты выглядят и ведут себя одинаково? Отклонение обосновано? |
| 4 | **Contrast** | Ключевые элементы выделены? «Delete» — красный, «Cancel» — серый? |
| 5 | **Proximity** | Связанные элементы рядом, несвязанные разделены? Play/Pause вместе, Quit — отдельно? |
| 6 | **Accessibility** | WCAG: контраст ≥ 4.5:1, alt text, keyboard nav, touch targets ≥ 44×44px? |
| 7 | **Alignment** | Сетка соблюдена? Элементы на одной линии? Выглядит «рассмотренным и намеренным»? |

### 4 совета

- **Perspective** — размещай элементы так, чтобы вести пользователя через логическую последовательность.
- **Effortless** — навигация консистентна, обратная связь очевидна, ярлыки (search) доступны.
- **Shortcuts** — часто используемые действия ускорены (keyboard shortcuts, quick actions).
- **Testing** — наблюдай, как люди используют интерфейс. Регулярно.

## Шаг 6. Использовать / построить дизайн-систему

Дизайн-система — источник правды для всей команды. Без неё consistency невозможна при масштабировании.

### 4 компонента дизайн-системы

| Компонент | Что включает | Зачем |
|-----------|-------------|-------|
| **UI kit** | Кнопки, формы, иконки, cards — reusable building blocks | Не перерисовывать одно и то же |
| **Design tokens** | Цвета, spacing, типографика — конкретные значения | Единый визуальный язык |
| **Patterns** | Поведение: навигация, модалки, формы, таблицы | Единый UX |
| **Documentation** | Как и когда использовать каждый элемент | Onboarding, consistency, governance |

### Когда строить, когда использовать существующую

| Ситуация | Рекомендация |
|----------|-------------|
| Стартап, MVP, маленькая команда | Возьми существующую (Material Design, Polaris, Carbon, Ant Design) и адаптируй |
| Средний продукт, растущая команда | Создай свою на основе существующей, с кастомными tokens |
| Enterprise, несколько продуктов | Полноценная кастомная система с governance и versioning |

### 6 выгод дизайн-системы

1. Ускоряет дизайн и разработку (до 34% по данным исследований).
2. Обеспечивает consistency.
3. Улучшает коллаборацию (дизайнеры + разработчики + PM из одного playbook).
4. Встраивает accessibility с самого начала.
5. Упрощает onboarding новых людей.
6. Масштабируется вместе с продуктом.

Подробнее: `references/design-system-guide.md`.

## Шаг 7. Валидировать: accessibility, этика, usability

Финальная проверка перед передачей в разработку.

### 7.1. Accessibility чеклист (WCAG 2.1)

| Область | Минимум |
|---------|---------|
| **Контраст** | Текст ≥ 4.5:1, крупный текст ≥ 3:1, UI-элементы ≥ 3:1 |
| **Keyboard** | Всё доступно с клавиатуры (tab, enter, escape) |
| **Screen reader** | Alt text для изображений, ARIA labels, semantic HTML |
| **Touch targets** | ≥ 44×44px (Apple HIG), ≥ 48×48dp (Material) |
| **Color** | Информация не только цветом (добавь иконку, текст, паттерн) |
| **Motion** | Respect `prefers-reduced-motion`; без мигающего контента |
| **Text** | Масштабируется до 200% без потери контента |
| **Forms** | Labels видимы, ошибки описательны, подсказки есть |

Инклюзивность — не фича, а фундамент. 15% населения имеют ту или иную форму ограничений.

Подробнее: `references/accessibility-checklist.md`.

### 7.2. Этический чеклист

- [ ] Нет тёмных паттернов (скрытые подписки, manipulation, disguised ads).
- [ ] Данные пользователя под его контролем (прозрачность, удаление).
- [ ] Контент инклюзивен (язык, изображения, представительство).
- [ ] Дизайн не эксплуатирует уязвимости (FOMO, addiction loops).

### 7.3. Usability тест

- Может ли пользователь пройти основной flow без инструкций?
- Где замедляется, ошибается, бросает?
- Стейкхолдеры дают фидбек по существу, а не по визуальным деталям?
- Если фидбек про «цвет» и «картинки» — вероятно, fidelity слишком высокая для текущего этапа.

## Шаг 8. Итерировать после запуска

UX-процесс не заканчивается релизом.

### Цикл

```
Launch → Analytics → User feedback → Prioritize → Redesign → Test → Launch
```

### Что отслеживать

| Метрика | Что показывает |
|---------|---------------|
| **Task completion rate** | Доля пользователей, завершивших ключевое действие |
| **Time on task** | Сколько времени уходит на задачу (меньше = лучше, обычно) |
| **Error rate** | Где ошибаются |
| **NPS / CSAT** | Общая удовлетворённость |
| **Bounce / drop-off** | Где уходят |
| **Retention** | Возвращаются ли |

### Design Thinking loop

**Empathize → Define → Ideate → Prototype & Test → Implement → (повторить)**

Процесс нелинеен: можно вернуться на любой шаг, если данные говорят, что гипотеза была неверна.

## UX vs UI — когда что

| | **UX** | **UI** |
|-|--------|--------|
| **Фокус** | Общий опыт, flow, исследование, тесты | Визуальный слой: кнопки, цвета, layout |
| **Вопрос** | «Понятно ли? Достижима ли цель?» | «Красиво ли? Консистентно ли?» |
| **Артефакты** | Personas, user flows, wireframes, test reports | Mockups, design system, UI kit, styleguide |
| **Метрики** | Completion rate, time on task, NPS | Visual consistency, brand alignment |

Оба нужны. UX без UI = функционально, но некрасиво. UI без UX = красиво, но непонятно.

## Interaction Design (IxD) — 5 измерений

Если задача затрагивает взаимодействие пользователя с элементами:

| # | Измерение | Что проверить |
|---|-----------|-------------|
| 1 | **Words** | Тексты кнопок, ошибок, подсказок — ясные, action-oriented? |
| 2 | **Visual representations** | Иконки, layout, типографика — поддерживают иерархию? |
| 3 | **Physical objects/space** | Учтено устройство (phone, tablet, desktop)? Ergonomics? |
| 4 | **Time** | Анимации, transitions, feedback — информативны, не декоративны? |
| 5 | **Behavior** | Что происходит при клике/свайпе/наведении — предсказуемо? |
