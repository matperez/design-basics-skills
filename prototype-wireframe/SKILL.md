# Prototype & Wireframe — процесс от идеи до валидированного прототипа

Этот скилл — пошаговый процесс: от формулировки гипотезы до тестируемого прототипа с реальными user flows. Не справочник «что такое wireframe», а дерево решений.

## Шаг 1. Определить, что валидируем

Прежде чем рисовать экраны, зафиксируй:

| Вопрос | Зачем | Пример |
|--------|-------|--------|
| **Что проверяем?** Новый продукт, фичу, поток, редизайн? | Определяет масштаб и fidelity | «Проверяем, понятен ли онбординг новым пользователям» |
| **Кто пользователь?** Персона, техническая грамотность, контекст | Влияет на сложность UI и flow | «Владелец малого бизнеса, 30–45, мобильный first» |
| **Какая цель пользователя?** Одна конкретная задача | Один flow = одна цель | «Создать и отправить счёт за 2 минуты» |
| **Где точка входа?** Поиск, push, главная, deep link? | Начало flow | «Push-уведомление → экран счёта» |
| **Что считаем успехом?** Подтверждение, dashboard, возврат? | Конец flow | «Экран "Счёт отправлен"» |
| **Какой вопрос к тесту?** Что узнаем из прототипа? | Фокусирует усилия | «Находят ли пользователи кнопку "Создать" без подсказки?» |

Результат: **гипотеза** в одном предложении: «Мы считаем, что [пользователь] сможет [действие] через [механику], и проверим это, наблюдая [метрику/поведение]».

## Шаг 2. Нарисовать user flow

User flow — визуальная карта: шаги пользователя от входа до цели. Рисуется **до** экранов.

### Процесс за 5 шагов

1. **Определи пользователя и цель** — кто, зачем, насколько tech-savvy.
2. **Зафиксируй точку входа** — откуда пользователь приходит.
3. **Нарисуй шаги** — значимые действия (формы, кнопки, переходы), не каждый клик.
4. **Добавь точки решений** — развилки (ошибка, альтернатива, возврат). Развилки = зоны риска фрустрации.
5. **Определи конечную точку** — подтверждение, dashboard, return.

### Правила user flow

- **Один flow = одна цель.** Не смешивай регистрацию и покупку.
- **Простота.** Слишком много ветвей = сложно и для команды, и для пользователя.
- **Консистентность обозначений.** Одни фигуры, цвета, подписи во всех flows.
- **Делись рано.** Фидбек на flow дешевле фикса после разработки.
- **Итерируй.** Flow — живой документ.

### Не путай диаграммы

| Диаграмма | Фокус | Когда |
|-----------|-------|-------|
| **User flow** | Шаги пользователя от A до B с развилками | Проектирование конкретного сценария |
| **Task flow** | Один сценарий без вариаций (прямая линия) | Простая задача, один путь |
| **Wireflow** | User flow + эскизы экранов | Нужно показать и путь, и UI |
| **User journey map** | Весь опыт: эмоции, мысли, touchpoints | Стратегический анализ всего опыта |
| **Sitemap** | Иерархия страниц (структура, не поведение) | Информационная архитектура |

Подробнее: `references/user-flow-guide.md`.

## Шаг 3. Выбрать уровень fidelity

Уровень детализации зависит от **вопроса, на который отвечаем**, а не от привычки.

### Матрица выбора

| Ситуация | Fidelity | Почему |
|----------|----------|--------|
| Ранняя гипотеза, спорная фича, brainstorm | **Low-fi** (бумага / digital wireframe) | Быстро, дёшево менять; фокус на структуре |
| Согласование потока и IA с командой | **Mid-fi** (wireframe + аннотации + реальный контент) | Команда видит логику; ещё нет «визуального шума» |
| Юзабилити-тест «как в продукте» | **High-fi** (цвет, типографика, интерактив) | Пользователь реагирует как на реальный продукт |
| Handoff в dev | **High-fi** (с состояниями и спеками) | Разработчик видит все edge cases |
| Есть дизайн-система + дизайн похож на существующие | **Сразу high-fi** | Разговор не уйдёт в визуал; команда привыкла к паттернам |

### 3 уровня wireframe

| Уровень | Что включает | Когда |
|---------|-------------|-------|
| **Low-fi** | Grayscale прямоугольники, lorem ipsum, базовая layout | «Это самое важное на экране?» |
| **Mid-fi** | Аннотации, реальный контент, ключевые взаимодействия | «Поток логичен? Контент на месте?» |
| **High-fi** | Бренд-шрифты, цвета, логотип (но без функциональности прототипа) | «Как это выглядит и ощущается?» |

### 3 типа прототипов

| Тип | Что | Плюсы | Минусы |
|-----|-----|-------|--------|
| **Бумажный** | Скетчи экранов, физическое переключение | Мгновенно, бесплатно, итерации за минуты | Сложно передать «ощущение продукта» |
| **Digital low-fi** | Wireframe + простые переходы | Проверка структуры и сценария; легко делиться | Нет «wow», пользователи могут не понять |
| **Digital high-fi** | Цвет, типографика, микровзаимодействия | Ближе к финальному опыту; мощнее для тестов | Дольше создавать; изменения дороже |

## Шаг 4. Построить wireframe

### Процесс

1. **Определи цели каждого экрана** — какое действие ведёт к успеху?
2. **Выбери размер** — Mobile: 393×852px, Tablet 11": 834×1194px, Desktop: 1440×1024px.
3. **Начни просто** — grayscale, минимум шрифтов, прямоугольники вместо картинок. Для контентных страниц — реальный текст, не lorem ipsum.
4. **Консистентность** — одинаковые компоненты выглядят одинаково на всех экранах.
5. **Навигация очевидна** — если нужна карта сайта, навигация требует доработки.
6. **Не привязывайся** — wireframe это черновик, а не финал.

### Чеклист готовности wireframe

- [ ] Все ключевые экраны нарисованы
- [ ] Навигационные пути определены и логичны
- [ ] Контент расставлен по приоритету (самое важное вверху/в центре)
- [ ] Интерактивные элементы обозначены
- [ ] Wireframe прошёл внутреннее ревью
- [ ] Учтены размеры под целевые устройства
- [ ] Готов к передаче в mockup / прототип

### Wireframe vs Mockup

| | Wireframe | Mockup |
|-|-----------|--------|
| **Фокус** | Структура, приоритеты, навигация | Визуальный стиль, бренд |
| **Детали** | Минимум | Цвета, шрифты, реальные изображения |
| **Обсуждаем** | «Что» и «куда» | «Как выглядит» |
| **Стоимость изменений** | Низкая | Выше |
| **Когда** | Раньше в процессе | Позже, когда поток утверждён |

Подробнее: `references/wireframing-guide.md`.

## Шаг 5. Превратить в прототип и тестировать

### Зачем прототипировать

1. **Валидация концепции** — стейкхолдеры видят «что строим» до разработки.
2. **Коммуникация** — общий язык для дизайна, бизнеса и dev.
3. **Обнаружение слепых зон** — юзабилити-тест выявит то, что не показали данные.

### Процесс rapid prototyping (цикл)

```
Build → Review → Refine → (повторить)
```

1. **Build** — создай mock-up потока (только релевантные экраны; не весь продукт).
2. **Review** — тест с целевыми пользователями + стейкхолдерами. Может ли пользователь пройти flow без инструкций?
3. **Refine** — измени по фидбеку. Повторяй, пока вопрос из шага 1 не получит ответ.

### Что включить в прототип

- **User flow** из шага 2 — каждый экран связан с предыдущим причинно.
- **Состояния** (для hi-fi): пустой, загрузка, ошибка, успех, disabled.
- **Обратная связь** — после CTA пользователь видит подтверждение.
- **Микровзаимодействия** (для hi-fi): hover, transitions, анимации.

### Как понять, что прототип «достаточно хорош»

- Пользователи проходят flow без подсказок.
- Стейкхолдеры дают фидбек по существу (структура, поток), а не по деталям визуала.
- Если фидбек уходит в «не нравится цвет» — понизь fidelity, убери визуальный шум.

## Шаг 6. Определить MVP scope (если это новый продукт)

MVP — не прототип. Это **реальный продукт с минимальным набором фич**, который выходит к пользователям.

### 4 признака MVP

1. **Affordable** — доступен для создания и повторения.
2. **Distills main idea** — ценность очевидна пользователю.
3. **Solves a problem** — добавляет реальную ценность.
4. **Adaptable** — архитектура позволяет расти.

### Как определить scope

Используй **MoSCoW** для приоритизации фич:

| Категория | Определение | Пример |
|-----------|------------|--------|
| **Must have** | Без этого продукт не работает | Регистрация, основной flow |
| **Should have** | Важно, но можно без них в v1 | Фильтры, настройки профиля |
| **Could have** | Приятно, но не критично | Темы оформления, gamification |
| **Won't have (this time)** | Осознанно откладываем | Интеграции, мультиязычность |

### Build — Measure — Learn (Lean Startup loop)

1. **Build** — создай MVP на основе гипотезы.
2. **Measure** — отслеживай метрики (используют ли фичу? как часто? как нашли?).
3. **Learn** — ответь честно: верна ли гипотеза? Что делать дальше?
4. Повтори.

**Явно фиксируй:** что входит в MVP и что **сознательно не входит** (и почему).

## Шаг 7. Дизайн приложения (если задача — полный цикл)

Если клиент строит приложение с нуля, процесс объединяет все шаги выше в 5 фаз:

| Фаза | Что делаем | Артефакты |
|------|-----------|-----------|
| 1. **Define** | Проблема, аудитория, рынок, персоны | Бриф, персоны, SWOT/competitive analysis |
| 2. **Design** | Wireframes, цвет, типографика, IA | Wireframes, style tile, sitemap |
| 3. **Prototype** | User flows → прототип → тест | User flows, prototype, test results |
| 4. **Develop** | Выбор типа (native, cross-platform, web, hybrid), dev handoff | PRD, specs, code |
| 5. **Launch & iterate** | Тесты, метрики, обратная связь, итерации | Analytics, release notes |

### Приоритизация фич приложения

Частые фичи мобильных приложений (не все нужны в v1):

- Push notifications
- GPS / location
- Платёжный flow
- Поиск и фильтры
- Соцсети-интеграции
- Трекинг заказов
- In-app поддержка
- Мультиязычность
- Рейтинги / отзывы
- Персонализация
- Gamification

Каждая фича проходит MoSCoW-фильтр: если не поддерживает главную задачу пользователя — не в v1.
