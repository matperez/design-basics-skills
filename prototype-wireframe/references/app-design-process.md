# Дизайн приложения: полный цикл

## Фаза 1: Define — определить идею

### Вопросы для старта

- Какую проблему решает приложение?
- Для кого оно? Что им важно?
- Что уже есть на рынке? Где пробел?
- Как приложение вписывается в миссию компании?

### Исследование рынка

- **Competitive analysis** — разобрать 3–5 конкурентов: фичи, UX, отзывы, слабые места.
- **SWOT** — strengths, weaknesses, opportunities, threats для своего продукта.
- **User interviews** — даже неформальные; ищи паттерны в том, как люди решают проблему сейчас.

### Персоны

Создай 1–3 персоны: имя, демография, цель, боль, контекст использования. Персоны = фильтр для всех дизайн-решений.

## Фаза 2: Design — спроектировать

### Wireframes

1. Определи ключевые экраны (обычно 5–10 для MVP).
2. Расставь контент по приоритету на каждом экране.
3. Проверь навигацию — пользователь должен знать, где он и куда идти.

### Цвет и типографика

- Примени процесс из скилла `color-theory` для палитры.
- Примени процесс из скилла `typography` для шрифтов.
- Пример: Calm = синие оттенки (спокойствие); DoorDash = красный (аппетит, срочность).

### Information architecture (IA)

| Принцип | Как применить |
|---------|--------------|
| **Organization** | Card sorting — как пользователи группируют контент? |
| **Labeling** | Понятные названия разделов и кнопок |
| **Navigation** | Меню, tabs, breadcrumbs — интуитивные |
| **Search** | Фильтры, подсказки, related results |

## Фаза 3: Prototype — построить и протестировать

### User flows

Для каждого ключевого сценария:
1. Точка входа → шаги → точки решений → конечная точка.
2. Один flow на одну задачу.

### Интерактивность

- Hover effects, transitions, микровзаимодействия.
- Пример: анимированное сердце при «лайке», смена цвета кнопки при hover.
- Цель: визуальная обратная связь, а не декорация.

### Тестирование

- **Moderated usability test** — может ли пользователь пройти flow без инструкций?
- **A/B test** — два варианта flow / UI → какой работает лучше.
- **Метрики:** время на задачу, completion rate, error rate, «где застревают».

## Фаза 4: Develop — выбрать стек и передать

### Типы приложений

| Тип | Плюсы | Минусы | Когда |
|-----|-------|--------|-------|
| **Native** (Swift/Kotlin) | Лучшая производительность, доступ ко всем API | Два кодовых бейса | Нужна максимальная скорость / hardware |
| **Cross-platform** (React Native, Flutter) | Один код → iOS + Android | Чуть ниже производительность | Хочется оба рынка с одной командой |
| **Web app** (PWA) | Без App Store, легко обновлять | Ограничен доступ к hardware | Простое приложение, быстрый запуск |
| **Hybrid** (Ionic, Capacitor) | Web-технологии внутри native-обёртки | Может быть медленнее | Контент-приложения, MVP |

### Dev handoff

- **PRD** (Product Requirements Document) — фичи, acceptance criteria.
- **Specs** — размеры, отступы, цвета, шрифты, состояния.
- **Assets** — иконки, иллюстрации в нужных форматах.
- **Prototype** — ссылка на интерактивный прототип как «живая документация».

## Фаза 5: Launch & Iterate

### До запуска

- [ ] QA: тесты на целевых устройствах и ОС.
- [ ] Accessibility: контраст, screen reader, touch targets (минимум 44×44px).
- [ ] Performance: время загрузки, размер бандла.
- [ ] Analytics: события для ключевых действий настроены.

### После запуска

- **Метрики:** DAU/MAU, retention, completion rate, NPS.
- **Фидбек:** отзывы в сторах, in-app surveys, поддержка.
- **Итерации:** Build → Measure → Learn → повторить.
