# Mobile Adaptation — процесс переноса десктопного макета на мобильные экраны

Этот скилл — пошаговый процесс: от аудита десктопного макета до готовой мобильной версии. Не справочник «что такое media query», а дерево решений, основанное на UX-исследованиях о мобильном поведении пользователей.

**Ключевой принцип:** мобильный ≠ «узкий десктоп». Это другой **контекст**, другое **намерение** и другие **ограничения** — поэтому задача не «сжать», а **пересобрать опыт**.

---

## Шаг 0. Понять мобильного пользователя

Прежде чем перекладывать блоки, зафиксируй три факта о мобильном контексте:

| Факт | Следствие для дизайна |
|------|----------------------|
| **Короткие сессии, прерывания** — средняя сессия ~70 секунд, 150 подходов в день (Google micro-moments) | Главный ответ и CTA — на первом экране; не рассчитывай на долгий скролл |
| **Одна рука, в движении** — точность касания ниже, ошибка дороже (лишний тап, потеря места) | Зоны касания ≥44×44px (WCAG AAA) / ≥24×24px (AA); частые действия — в нижней трети экрана (thumb zone) |
| **Когнитивная перегрузка** — на узком экране меньше контекста, рабочая память загружена сильнее | Один фокус на скролл-экран; progressive disclosure для второстепенного; чёткие границы секций |

**Подробнее:** `references/mobile-user-context.md`

## Шаг 1. Аудит десктопного макета

Составь **инвентаризацию блоков** десктопной страницы: перечисли все секции по порядку.

Для каждого блока определи:

| Характеристика | Что записать |
|----------------|-------------|
| **Название блока** | Например: Hero, Services grid, About, FAQ, Footer |
| **Тип контента** | Текст, изображение, форма, карточки, таблица, видео |
| **Роль для конверсии** | Первичная (CTA, value proposition), поддерживающая (доверие, социальное доказательство), вторичная (контекст, детали) |
| **Зависимость от ширины** | Многоколоночный grid, боковая колонка, широкая таблица, hero с горизонтальным визуалом |

Результат: **таблица блоков** с колонками: порядок на десктопе → роль → проблема на 375px.

## Шаг 2. Приоритизировать контент для мобильного сценария

Для каждого блока из таблицы присвой **мобильный приоритет** по трём категориям:

### Категория A: оставить видимым

Блоки, которые закрывают **основное намерение** мобильного посетителя.

| Тип | Почему |
|-----|--------|
| **Ценностное предложение / ответ** | Ради этого открыли сайт в микромоменте |
| **Основной CTA** | Путь от намерения к действию должен быть кратчайшим |
| **Краткое доверие рядом с решением** | Без него не нажмут: 1–2 отзыва, логотипы, «10 лет опыта» |
| **Навигация к ключевым разделам** | Не вся структура, но выходы на топ-задачи |
| **Контакт** | Сильно коррелирует с «I-want-to-go» / «I-want-to-know» |

### Категория B: свернуть (progressive disclosure)

Блоки, которые **нужны части аудитории**, но не большинству в типичной мобильной сессии.

| Тип | Паттерн |
|-----|---------|
| FAQ с 5+ вопросами | Аккордеон; показать 2–3 первых, остальные за «Ещё N вопросов» |
| Подробные характеристики | Раскрываемый блок «Подробнее» |
| Длинные списки услуг | Показать 3–4 карточки + «Все услуги» |
| Детальные биографии, кейсы | Краткое резюме + ссылка / раскрытие |

**Правило:** свёрнутый контент **остаётся в DOM** — это важно для SEO (mobile-first indexing) и доступности. Скрываем **визуально**, а не удаляем.

### Категория C: убрать или переместить вниз

Блоки, которые **шумят**, **дублируют** или существуют для красоты десктопа.

| Тип | Почему |
|-----|--------|
| Повторяющийся маркетинг («мы лучшие» x3) | На мобиле воспринимается как навязывание |
| Боковые колонки с доп. ссылками | Перебивают поток; переместить в конец или в меню |
| Декоративные визуалы без функции | Конкурируют с полезным за скорость и экран |
| Второй и третий равнозначные CTA рядом | Размывают выбор; оставить один главный, остальное — текстовые ссылки |
| Тяжёлые фоновые видео | Трафик + батарея; заменить на статичное изображение |

**Важно:** «убрать с первого экрана» ≠ «удалить совсем». Если контент ценен для SEO или части аудитории — **переместить ниже** или за раскрытие, а не стирать.

## Шаг 3. Спланировать мобильный порядок секций

Перестройка порядка блоков — **нормальная часть адаптации**, не «предательство макета».

### Принципы сортировки

1. **Первый экран (≈667px высоты):** ценность + CTA + минимум доверия. Без «пролога».
2. **Далее:** раскрытие → доказательства → детали → финальный CTA.
3. **Порядок определяется задачей посетителя**, а не десктопной композицией.

### Шаблон мобильного порядка (для лендинга)

```
1. Header (лого + гамбургер + 1 кнопка)
2. Hero (заголовок + подзаголовок + CTA) — без тяжёлого фонового визуала
3. Доверие-строка (3–5 логотипов или цифр)
4. Услуги / Benefits (вертикальный список карточек, одна колонка)
5. О компании (сокращённый текст + «Подробнее»)
6. Отзывы (1–2 + «Все отзывы»)
7. FAQ (аккордеон, 3–5 вопросов)
8. Финальный CTA (повтор кнопки с усилением)
9. Footer (контакт, ссылки, юр. инфо)
```

## Шаг 4. Адаптировать компоненты

### Навигация

| Десктоп | Мобильное решение |
|---------|------------------|
| Горизонтальное меню 5–8 пунктов | Гамбургер (☰) + 1 основной CTA-кнопка видимая |
| Mega-menu | Вертикальный список с аккордеонами |
| Sidebar навигация | Убрать; ключевые ссылки — в гамбургер или footer |
| Sticky top bar | Допустимо, но **компактный**: лого + гамбургер + CTA, высота ≤60px |

### Сетка

| Десктоп | Мобильное решение |
|---------|------------------|
| 3–4 колонки карточек | **1 колонка**; при ≥500px — 2 колонки |
| Двухколоночный split | Вертикальный стек: визуал сверху, текст снизу |
| Таблица с 6+ столбцами | Горизонтальный скролл **с подсказкой** или карточное представление |

### Типографика

| Параметр | Рекомендация |
|----------|-------------|
| Базовый размер текста | ≥16px (чтобы iOS не зумил формы) |
| Заголовок hero | Fluid: `clamp(1.75rem, 5vw, 3.5rem)` |
| Длина строки | 35–55 символов на мобиле (не ≥80 как на десктопе) |
| Межстрочный интервал | 1.4–1.6 для body text |

### Изображения

| Правило | Как |
|---------|-----|
| Responsive source | `srcset` + `sizes`; WebP/AVIF где возможно |
| Aspect ratio без CLS | Явные `width`/`height` или CSS `aspect-ratio` |
| Lazy loading | `loading="lazy"` для всего ниже fold |
| Hero-изображение | Отдельный кроп для мобилки (вертикальный или квадратный), а не горизонтальный десктопный |

### Кнопки и интерактивные элементы

| Правило | Значение |
|---------|---------|
| Минимальная зона касания | 44×44px (цель WCAG AAA); абсолютный минимум 24×24px (AA) |
| Расстояние между кликабельными зонами | ≥8px |
| Основной CTA | Полная ширина (`width: 100%`) на мобиле; визуально выделен |
| Ссылки в тексте | Достаточный padding для тапа; не стоит полагаться на точное касание слова |

## Шаг 5. Проверить результат

### Чеклист мобильной адаптации

**Структура и контент:**
- [ ] Первый экран содержит ценность + CTA (не «пролог»)
- [ ] Порядок секций соответствует мобильному сценарию (не дублирует десктоп бездумно)
- [ ] Второстепенный контент свёрнут, а не удалён
- [ ] Весь текст контента доступен (для SEO и a11y) — не `display:none` без альтернативы

**Навигация:**
- [ ] Меню сжато (гамбургер или bottom bar)
- [ ] CTA видим без открытия меню
- [ ] Footer содержит ключевые ссылки и контакт

**Касания и взаимодействия:**
- [ ] Все кнопки ≥44×44px
- [ ] Между кликабельными зонами ≥8px
- [ ] Формы: поля не обрезаются клавиатурой, labels видимы

**Визуал и производительность:**
- [ ] Изображения используют responsive source; нет десктопных 2000px на мобиле
- [ ] Hero-изображение кадрировано для вертикали
- [ ] Нет горизонтального скролла (кроме осознанных таблиц с подсказкой)
- [ ] Шрифт ≥16px для body; заголовки fluid

**Тестирование:**
- [ ] Проверено на ширине 375px и 390px
- [ ] Проверено с увеличенным шрифтом ОС (zoom 200%)
- [ ] Проверено `prefers-reduced-motion` (анимации не критичны)
- [ ] На реальном устройстве (если возможно) — не только эмулятор

## Антипаттерны: чего не делать

| Антипаттерн | Почему плохо | Что вместо |
|-------------|-------------|------------|
| Скопировать десктопный порядок секций 1:1 | На мобиле «сюжет» другой; пролог раздражает | Пересортировать по мобильному намерению |
| Скрыть CTA за гамбургером | Главное действие теряется; конверсия падает | CTA всегда видим в header или sticky bar |
| `display: none` на весь блок без альтернативы | Потеря контента для SEO и screen readers | Свернуть (аккордеон) или переместить вниз |
| Горизонтальная навигация из 7 пунктов | Не помещается; ломает строку | Гамбургер + 1–2 видимых ключевых |
| Один и тот же hero-кроп (landscape) | На вертикальном экране лица обрезаны, текст нечитаем | Отдельный мобильный кроп (portrait/square) |
| Фиксированные ширины в px для колонок | Горизонтальный скролл | `%`, `fr`, `minmax()`, `clamp()` |
| Попытка pixel-perfect iOS ↔ Android | Шрифты, safe areas, скролл различаются | Целиться в **предсказуемость**, не идентичность |
