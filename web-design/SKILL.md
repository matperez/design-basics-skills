# Web Design — процесс от стратегии до запуска сайта

Этот скилл — пошаговый процесс: от определения целей сайта до responsive, конверсионного, SEO-готового продукта. Не справочник «что такое grid», а дерево решений.

---

## Шаг 1. Определить стратегию

Прежде чем рисовать, ответь на 6 вопросов:

| Вопрос | Зачем | Пример |
|--------|-------|--------|
| **Цель сайта** — что должен сделать посетитель? | Определяет всё: структуру, CTA, контент | «Оставить заявку на консультацию» |
| **Аудитория** — кто, откуда, с какого устройства? | Влияет на тон, сложность, mobile-first | «Владельцы малого бизнеса, 80% мобильный трафик» |
| **Тип сайта** — лендинг, портфолио, магазин, блог, SaaS? | Определяет структуру и паттерны | «E-commerce с каталогом 500+ товаров» |
| **Контент** — что уже есть, что создавать? | Контент первичен, дизайн обслуживает | «Есть тексты и фото; нужны видео-обзоры» |
| **Конкуренты** — как выглядят, что работает? | Найти пространство для отстройки | «Все конкуренты минималистичные — мы можем быть ярче» |
| **Ограничения** — бюджет, сроки, стек, CMS? | Реалистичный scope | «WordPress, 4 недели, 1 дизайнер» |

### Web design vs Web development

| | **Дизайн** | **Разработка** |
|-|-----------|---------------|
| **Фокус** | UX/UI, визуал, шаблоны, wireframes, прототипы | HTML/CSS/JS, бэкенд, CMS, API |
| **Артефакты** | Mockups, style guide, design system, prototype | Код, деплой, тесты, CI/CD |
| **Навыки** | Композиция, цвет, типографика, UX research | Программирование, базы данных, DevOps |

Оба нужны: дизайн без разработки = картинки, разработка без дизайна = функциональный, но непривлекательный продукт.

## Шаг 2. Выбрать структуру сайта

Структура = как страницы связаны друг с другом. Влияет на навигацию, SEO, масштабируемость.

| Структура | Что | Когда | Пример |
|-----------|-----|-------|--------|
| **Hierarchical (tree)** | Главная → категории → подкатегории | E-commerce, бизнес-сайты, информационные | Amazon, корпоративные сайты |
| **Linear (sequential)** | Шаг 1 → Шаг 2 → Шаг 3 | Онбординг, туториалы, multi-step forms | Курсы, sign-up flow |
| **Matrix (web)** | Страницы связаны сетью ссылок | Wiki, базы знаний, портфолио | Wikipedia, personal sites |
| **Database (dynamic)** | Страницы генерируются из данных по запросам | Большие каталоги, новостные сайты | E-commerce с фильтрами, CMS |
| **Hybrid** | Комбинация двух и более | Крупные сайты с разными задачами | Университеты (курсы + ресурсы) |

**Если не уверен:** hierarchical — самый безопасный дефолт. Большинство сайтов начинают с него.

### Компоненты структуры

- **Sitemap** — иерархия страниц (нарисуй до дизайна).
- **URL-структура** — чистая, читаемая, отражающая иерархию (`/products/shoes/running`).
- **Internal linking** — связи между страницами (помогает SEO и навигации).
- **Breadcrumbs** — «хлебные крошки» показывают позицию в иерархии.

Подробнее: `references/website-structure-guide.md`.

## Шаг 3. Спланировать layout и grid

### Выбрать grid-систему

Grid — невидимый каркас, на котором держится layout.

| Тип grid | Что | Когда |
|----------|-----|-------|
| **Block (single-column)** | Один столбец, вертикальное чтение | Блоги, статьи, лонгриды |
| **Column grid** (2–12 колонок) | Вертикальные колонки для организации | Прайсинг, сравнение, новостные сайты |
| **Modular grid** | Колонки + строки = модули | Галереи, e-commerce каталоги, Instagram-like |
| **Hierarchical grid** | Свободное размещение по визуальной важности | Креативные сайты, журнальные layout |
| **Baseline grid** | Горизонтальные линии для выравнивания текста | Типографически насыщенные страницы |

### Элементы grid

| Элемент | Что делает |
|---------|-----------|
| **Margins** | Отступы от края экрана |
| **Columns** | Вертикальные блоки для контента |
| **Gutters** | Промежутки между колонками |
| **Rows / flowlines** | Горизонтальные разделители секций |
| **Modules** | Пересечение колонок и строк = ячейки контента |

### Единицы измерения

- **Base unit** — 8px (или 4px); все размеры кратны ему.
- **REM** > px для responsive: масштабируется с настройками пользователя.
- **Padding** — пространство внутри элемента; измеряется в base units.

### Паттерны сканирования

| Паттерн | Как работает | Когда |
|---------|-------------|-------|
| **Z-pattern** | Взгляд по Z: верхний левый → верхний правый → диагональ → нижний правый | Лендинги, минималистичные страницы |
| **F-pattern** | Горизонтально сверху, потом вниз по левому краю | Текстовые страницы, блоги, поисковая выдача |
| **Upper-left gravity** | Взгляд начинается с верхнего левого угла | Почти все — поэтому лого и nav слева/сверху |

### 12 типов layout

| Layout | Best for |
|--------|---------|
| **Grid** | E-commerce, блоги, responsive |
| **Split-screen** | Лендинги, сравнение, sign-up |
| **Asymmetrical** | Продуктовые страницы, акцент на одной зоне |
| **Full-screen** | Hero-секции, минималистичные лендинги |
| **Side-scrolling** | Галереи, категории (Netflix-style) |
| **Card** | Каталоги, dashboard, Pinterest-like |
| **Magazine** | Новостные сайты, медиа |
| **Single-column** | Блоги, лонгриды |
| **Hero** | Лендинги, портфолио (большой визуал + CTA) |
| **Sidebar** | Блоги, документация, dashboards |
| **Alternating** | Лендинги (секция слева ↔ секция справа) |
| **Featured image** | Портфолио, фотография |

Подробнее: `references/layout-catalog.md`.

## Шаг 4. Спроектировать навигацию и контент

### Навигация — паттерны

| Паттерн | Плюсы | Минусы | Когда |
|---------|-------|--------|-------|
| **Top bar (horizontal nav)** | Видима, знакома | Ограничена по количеству пунктов | Десктоп, до 7 пунктов |
| **Sidebar** | Вмещает много пунктов, иерархична | Занимает горизонталь | Dashboard, документация, admin |
| **Hamburger menu** | Экономит место, чистый интерфейс | Скрывает навигацию → ниже CTR | Mobile-first, минимализм |
| **Bottom tab bar** | Thumb zone, всегда видима | 3–5 пунктов максимум | Мобильные приложения |
| **Breadcrumbs** | Показывает позицию в иерархии | Не основная навигация | Глубокие иерархии, e-commerce |
| **Mega menu** | Показывает всю структуру сразу | Может перегрузить | Крупные e-commerce, enterprise |

**Hamburger menu — когда использовать:**
- Mobile-first, ограниченное пространство.
- Вторичная навигация.
- Пользователи знакомы с паттерном.

**Когда избегать:**
- Критически важные страницы должны быть видимы.
- Сложная навигация → скрытие = потерянные пользователи.

### Контент — принципы

- **Контент первичен** — определи, что пользователю нужно знать/сделать, потом проектируй вокруг этого.
- **Заголовки** — сканируемые, информативные, отражают содержание.
- **CTA (Call-to-Action)** — на каждой странице ясно, что делать дальше.
- **Белое пространство** — не бойся пустоты; она направляет внимание.
- **Реальный контент** > Lorem ipsum (особенно для контентных страниц).

### CTA — правила

| Правило | Пример |
|---------|--------|
| **Одно главное действие на экран** | «Get started for free» |
| **Язык = ценность, не инструкция** | «Save 20% today» > «Click here» |
| **Контраст** — CTA выделяется визуально | Яркая кнопка на нейтральном фоне |
| **Placement** — above the fold + повтор ниже | Netflix: sticky CTA при скролле |
| **Urgency без манипуляции** | «Free trial — no credit card needed» |
| **Progression** — CTA усложняется по мере скролла | «Learn more» → «Start free trial» → «Buy now» |

Подробнее: `references/cta-playbook.md`.

## Шаг 5. Применить визуальный стиль

### 5 столпов визуального стиля

| # | Столп | Что проверить |
|---|-------|-------------|
| 1 | **Цветовая схема** | Палитра поддерживает бренд и иерархию? 60-30-10? Контраст WCAG? (Скилл `color-theory`) |
| 2 | **Типографика** | Читаема на всех экранах? Иерархия (h1–h6)? 1–2 шрифта? (Скилл `typography`) |
| 3 | **Иерархия** | Самое важное видно первым? Size, color, contrast, proximity работают вместе? |
| 4 | **Баланс и ритм** | Сетка соблюдена? White space достаточно? Элементы не «толпятся»? |
| 5 | **Consistency** | Одни и те же компоненты выглядят одинаково? Style guide соблюдается? |

### Цветовые схемы для сайтов

| Тип | Когда | Примеры |
|-----|-------|---------|
| **Minimal / neutral** | Elegance, фокус, чистота | Asana, Calm — whites, grays, soft pastels |
| **Bold / vibrant** | Энергия, молодость, action | Spotify, Slack — яркие акценты |
| **Dark** | Премиальность, контраст, медиа | Netflix, Steam — тёмный фон, яркие акценты |
| **Earthy / warm** | Природа, sustainability, handmade | Organic brands — browns, greens, beiges |
| **Pastel** | Мягкость, забота, wellness | Healthcare, baby products — soft pinks, blues |
| **Monochromatic** | Простота, графичность | Portfolios, editorial — один hue, много тонов |

## Шаг 6. Сделать responsive и mobile-first

### Mobile-first → responsive

Начни с маленького экрана, расширяй media queries вверх.

### 7 принципов mobile-first

1. **Content inventory** — определи, что нужно на ходу. Аудит → приоритизация.
2. **Сначала маленький экран** — 320px → 768px → 1024px → 1440px.
3. **Упрощённая навигация** — hamburger, bottom tab bar, thumb zone.
4. **Touch-friendly** — кнопки ≥ 44×44px, достаточные отступы.
5. **Performance** — сжатие изображений, минимум скриптов, lazy loading.
6. **Fluid typography** — масштабируемые шрифты и spacing.
7. **Тестирование на реальных устройствах** — симуляторы ≠ реальность.

### Breakpoints (ориентиры)

| Устройство | Ширина |
|-----------|--------|
| Mobile (small) | 320–375px |
| Mobile (medium) | 376–428px |
| Tablet | 768–834px |
| Laptop | 1024–1440px |
| Desktop (large) | 1440px+ |

### Responsive-компоненты

- **Flexible grid** — колонки в процентах, не в пикселях.
- **Media queries** — CSS-правила по breakpoints.
- **Responsive images** — `srcset`, `max-width: 100%`, WebP/AVIF.
- **Fluid typography** — `clamp()` для font-size.

Подробнее: `references/responsive-and-mobile-first.md`.

## Шаг 7. Лендинги (если задача — конверсия)

Лендинг = одна страница, одна цель, один CTA.

### Структура лендинга

| Секция | Что | Зачем |
|--------|-----|-------|
| **Hero** | Заголовок + value proposition + CTA + визуал | Первые 50ms — решают, останется ли пользователь |
| **Social proof** | Отзывы, логотипы клиентов, цифры | Доверие |
| **Features / Benefits** | Что получает пользователь | Раскрытие ценности |
| **How it works** | 3–5 шагов | Снижение неопределённости |
| **Pricing** (опционально) | Тарифы, сравнение | Принятие решения |
| **FAQ** | Ответы на возражения | Снятие барьеров |
| **Final CTA** | Повтор CTA с усилением | Последний шанс конвертировать |

### Типы лендингов

| Тип | Цель | Пример |
|-----|------|--------|
| **Product launch** | Генерация interest / preorder | Apple product pages |
| **Lead generation** | Сбор контактов | «Download free guide» |
| **SaaS signup** | Регистрация / free trial | Figma, Notion, Slack |
| **E-commerce** | Продажа товара | Product pages |
| **Event** | Регистрация на мероприятие | Conference sites |
| **Portfolio** | Демонстрация работ | Personal sites |

## Шаг 8. Проверить и запустить

### Чеклист перед запуском

**Performance:**
- [ ] Время загрузки < 3 секунды (целевое).
- [ ] Изображения оптимизированы (WebP/AVIF, lazy loading).
- [ ] Минимум блокирующих скриптов.

**SEO:**
- [ ] Семантический HTML (`<header>`, `<nav>`, `<main>`, `<article>`, `<footer>`).
- [ ] Meta title + description на каждой странице.
- [ ] Alt text для изображений.
- [ ] Sitemap.xml + robots.txt.
- [ ] Мобильная версия индексируема (Google mobile-first indexing).

**Accessibility:**
- [ ] Контраст ≥ 4.5:1 (текст), ≥ 3:1 (UI).
- [ ] Keyboard navigation работает.
- [ ] Screen reader — проверено.
- [ ] Touch targets ≥ 44×44px.

**UX:**
- [ ] Навигация работает на всех устройствах.
- [ ] CTA заметны и понятны.
- [ ] Формы — с labels, error messages, валидацией.
- [ ] 404 страница — полезна (ссылки, поиск).

**Cross-browser / cross-device:**
- [ ] Chrome, Safari, Firefox, Edge.
- [ ] iOS Safari, Android Chrome.
- [ ] Разные разрешения (от 320px до 2560px).

### Когда нужен редизайн (а не refresh)

| Сигнал | Что делать |
|--------|-----------|
| Устаревший дизайн | Redesign: структура + визуал + контент |
| Плохой UX (высокий bounce) | Redesign: IA + навигация + flows |
| Падение метрик | Audit → определить причину → redesign или refresh |
| Проблемы с производительностью | Может быть technical debt, не дизайн |
| Security vulnerabilities | Development fix, не дизайн (но может потребовать redesign при смене стека) |

**Redesign vs Refresh:**
- **Refresh** — обновление визуала (цвета, шрифты, изображения) без изменения структуры.
- **Redesign** — фундаментальное изменение: структура, IA, навигация, контент, дизайн.

### Итерации после запуска

```
Launch → Analytics → User feedback → Prioritize → Improve → Test → Deploy
```

Сайт — живой продукт. Метрики (bounce rate, conversion, time on page) показывают, что работает.
